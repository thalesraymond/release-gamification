## 2024-05-14 - Initial Setup
**Discovery:** The development environment uses a Docker-based MongoDB instance with persistent volume storage for local development, rather than `mongodb-memory-server` which was previously used. This was done to ensure data persists across restarts. The project is a monorepo using pnpm. The `packages/infrastructure` package uses Vitest for testing. The main backend API (`apps/apis`) is built with Fastify. OpenSpec is used for architectural change proposals.
**Impact:** Future proposals should assume a Fastify + MongoDB stack with Clean Architecture. Proposals should be created using `npx @fission-ai/openspec new change "<name>"`.
**Source:** `openspec/changes/use-docker-mongodb/proposal.md`, Memory
## 2026-03-21 - External System Data Ingestion
**Discovery:** The domain model includes entities for external systems, such as `DeliveryServiceNowChangeDetails` and `DeliveryGithubIssueDetails`. While a GitHub webhook API was implemented for ingesting GitHub issues/PRs, there is currently no mechanism to ingest ServiceNow changes. External system data ingestion relies on a webhook-driven architecture within the Fastify API (e.g., `apps/apis/src/routes/github-webhooks.ts`).
**Impact:** Future features needing external system synchronization should follow the established webhook pattern. A logical next step is implementing a ServiceNow webhook API to populate the existing ServiceNow domain models.
**Source:** `packages/domain/src/DeliveryServiceNowChangeDetails.ts`, `apps/apis/src/routes/github-webhooks.ts`, `openspec/changes/archive/2026-03-19-github-webhook-api/proposal.md`
